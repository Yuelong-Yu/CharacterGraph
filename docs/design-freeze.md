## 1. 产品定位

一个**桌面专属、中文界面**的希腊神话人物关系图谱网站。访客以 Node + Edge 形式探索 ~150 位核心人物及其关系，点击节点查看人物详情，点击边查看双人之间的事件时间线。

视觉风格参考 Dota 英雄选择 + 三国杀卡牌——深色基底、AI 生成的半身立绘、戏剧化打光。

## 2. 内容范围

| 项 | 决策 |
|---|---|
| 人物规模 | **150 核心人物** |
| 数据来源 | **双源 Wikipedia（中+英）** |
| 抓取策略 | 抓 Wiki 索引 → 按英文 Wiki 月均 pageview 排序 → 取前 150 深抓 |
| 字段缺省 | **允许"史料无记载"**，宁缺毋滥不编造 |
| 产品语言 | **纯中文**（英文 Wiki 为主源 + 中文 Wiki 锁译名 + LLM 翻译，需维护术语表） |

### 2.1 人物 7 字段

1. 生平
2. 主要事件
3. **名言**（仅含古典文献出处的原句，标注《伊利亚特》卷 X / Odyssey X.Y，中文译本对照——译者选择实施阶段决定，建议罗念生/王焕生）
4. 武器
5. 技能（动词短语，与神职区分）
6. **神职/领域**（与"技能"并列的独立字段）
7. 坐骑

### 2.3 关系 5 类（边颜色）

| 类 | 颜色 |
|---|---|
| 血缘 | 暖白 |
| 婚姻/情人 | 玫瑰金 |
| 敌对 | 暗红 |
| 同伴/战友 | 青蓝 |

## 3. 视觉与图像

| 项 | 决策 |
|---|---|
| 整体风格 | Dota + 三国杀卡牌风，深色基底 |
| 图像数量 | 150 张（每人一张，半身含头部） |
| 画幅 | **2:3 竖向，800×1200** |
| 节点缩略 | 构建期自动裁中心 1:1 → 128×128 头部 |
| 构图 | 半身（腰部以上），手部不入画（规避 AI 手部畸形） |
| 背景 | 深色渐变 + 极简符号，不画环境 |
| 打光 | 侧逆光戏剧化（chiaroscuro） |
| 边框/装饰 | **图像本身不画边框**，前端 CSS 叠加 |
| 着装策略 | **(ii) 艺术裸体允许**——合规风险已签收，不在中国大陆上线 |
| 风格统一策略 | **(B) 子风格族** + 全局基底 prompt + 5 类变量 prompt |
| 原始神画法 | **(y) 抽象化**（星云/漩涡/虚空），共享同 sref 保持家族感 |

### 3.1 图像生成

- **工具**：`doubao-seedream-5.0-lite`（火山方舟 Agent Plan）
- **API key**：用户提供的 key，写入本地 `.env`（加入 `.gitignore`），代码用 `os.getenv("ARK_API_KEY")` 引用
- **计费**：包月套餐
- **候选策略**：**(A) 一次过**，每人 1 张
- **生成顺序**：**18 → 30 → 100 三批**，18 人完成后停下、跑通整站、用户验收，再扩展
- **健康检查**：先发一个最小请求验证模型存在性、参数兼容性、URL 时效（火山方舟图片 URL 有过期时间，需下载落盘）

### 3.2 字体

- 标题：Cormorant Garamond（开源衬线）
- 中文正文：思源宋体（与衬线英文匹配）
- 数字/标签/出处：JetBrains Mono / IBM Plex Mono

## 4. 图谱布局

| 项 | 决策 |
|---|---|
| 布局算法 | **(B) 族谱为骨、关系为肌** |
| 默认视图 | 仅显示血缘边（dagre 层级），节点按代际分层 |
| 工具栏开关 | 显示婚姻 / 敌对 / 同伴 / 师徒（访客按需点亮） |
| 节点坐标 | **一次锁定**——dagre 初始布局 → 手工微调 → 存 JSON |
| 节点形状 | **(i) 统一圆角矩形卡片**，所有信息靠边框颜色和内部内容承载 |

### 4.1 dagre 环处理

- dagre 不支持环——俄狄浦斯-伊俄卡斯特、宙斯-赫拉这类复合关系：保留主关系入 dagre，副关系作为悬浮注释

## 5. 交互

### 5.1 首屏

- **(B) 代际入场动画 + (E) 教学**：3-5 秒动画从混沌/盖亚开始，按代际逐层展开
- 动画过程中点出 3 个核心编码：节点边框=类别 / 边颜色=关系 / 节点卡片有头像和称号
- **localStorage 记忆"已看过"**，回访直接进入静态图谱
- 动画期间并行加载头像图片

### 5.2 详情面板

- **(a) 同一面板互斥切换**——点节点显示人物模式，点边显示关系模式，切换用 200ms slide 转场
- **(D) 上下分区**：
  - 上：固定身份卡（头像 + 名字 + 称号 + 时代 + 武器/坐骑图标）
  - 下：滚动长内容（生平 + 主要事件 + 名言）
- 顶部面包屑"← 返回 XXX"
- **人名自动 hyperlink**（用术语表锁定），点击切换到对应人物

### 5.3 边详情（关系模式）

- 双人头像 + 关系类型徽章（5 类之一） + 复合标记
- **(β) 事件时间线**：每个事件卡 = 事件名 + 100-200 字描述 + 出处 + 相关人物链接 + 可展开 300-500 字详版
- 时间线按神话内时间排序

### 5.4 搜索

- **(B+C) 模糊名+别名+全文**
- 顶部固定搜索框
- 输入支持：中文译名 / 英文原名 / 别名（"朱庇特"命中宙斯）
- 全文索引：fuse.js 客户端搜索 150 人物的"生平+事件"文本
- 命中后：节点高亮 + 视图自动平移居中
- fuse.js 用 dynamic import 推迟到搜索框聚焦时加载

### 5.5 图例

- 右上角 i 图标 → 折叠图例面板
- 列出 10 类节点 + 5 类边 + 边粗细含义

### 5.6 移动端

- **放弃移动端**——< 1024px 显示"请用电脑访问"提示页

## 8. 数据 Schema (Zod)

### 8.1 Character

```ts
{
  schema_version: 1,
  id: string,                    // slug, e.g. "zeus"
  name_zh: string,               // 主中文名
  name_en: string,
  aliases: string[],             // 别名（朱庇特、雷霆神等）
  category: enum(10类),          // 决定边框色
  era_layer: int,                // 代际，0=原始神 ... 5=凡人/英雄
  bio: string | null,
  events: { title, desc, source }[] | [],
  quotes: { text, source }[] | [],     // 含古典文献出处
  weapons: string[] | [],
  skills: string[] | [],
  domains: string[] | [],
  mounts: string[] | [],
  portrait: string,              // 图片路径
  thumb: string                  // 缩略路径
}
```

### 8.2 Relation (Edge)

```ts
{
  schema_version: 1,
  id: string,
  source: character.id,
  target: character.id,
  primary_type: enum(5类),       // 决定边色
  composite_types: enum[] | [],  // 复合关系
  events: { title, desc, source, era_order }[],   // 事件链时间线
  event_count: int               // 决定边粗细
}
```

## 9. 性能预算

| 指标 | 目标 |
|---|---|
| LCP | < 2.5s |
| INP | < 200ms |
| CLS | < 0.1 |
| JS bundle (gz) | 目标 100-150KB（含 React+React Flow） |
| CSS | < 30KB |

### 9.1 性能策略：A 三层金字塔 + D Service Worker

1. **首屏** (< 1.5s)：bundle + 数据 JSON + SVG 占位符图谱（空卡片+边框色，无头像）
2. **第二屏** (~3s)：入场动画播放，期间懒加载头部缩略图，动画结束时显示真实头像
3. **按需加载**：点击节点 → 加载半身大图 + 完整字段
4. **Service Worker**：缓存所有图像，二访秒开

## 12. 实施顺序

实施阶段按以下顺序，**每完成一步停下汇报，不一上来烧 API**：

1. 初始化仓库（pnpm + uv 双轨）+ `.gitignore` + Zod schema 文件
2. 数据 pipeline：抓 18 人 Wiki → LLM 结构化 → 产出 18 个 character JSON + 关系 JSON
3. Next.js 骨架：双栏布局 + React Flow + 占位符节点（无图）
4. 前端组件：节点卡片、详情面板、边时间线、搜索、图例
5. 入场动画
6. doubao-seedream 健康检查脚本（先 1 张试水）
7. 图像 pipeline：跑 18 张半身像 + 缩略生成
8. 整合 + 性能优化（Service Worker、懒加载、bundle 分析）
9. **本地验收 milestone**——用户确认通过后才进入第 2 批 30 人
